HexHawk features

Every module answers: what goes in, what comes out, when do I use it, and what did it not prove?

Consumer-friendly technical tools explain their inputs, workflows, outputs, and limits. HexHawk uses the same plain-language pattern: start with file facts, inspect code and evidence, keep helper outputs labelled, and export a report another analyst can check.

NEW USER MAP

If you remember only one thing, remember the handoff.

HexHawk is not just a place to look at bytes. It is a way to hand off a binary investigation with identity, evidence, verdict authority, helper labels, and uncertainty intact.

START WITH FACTS

Begin with evidence you can point to.

Available

File identity

Use it when: you first open a binary. Input: a local file path. Output: hashes, size, type, sections, imports, strings, entropy clues, and architecture hints. Limit: file facts do not prove intent by themselves.

Available

Disassembly and graph clues

Use it when: you need to see instruction flow. Input: parsed code bytes. Output: code views, branches, xrefs, and graph clues. Limit: graph shape is evidence to review, not a verdict.

Available

ProgramAnalysis

Use it when: you need one model of functions, calls, references, imports, and CFG context. Output: structured evidence used by UI panels and exports. Limit: it organizes analysis data; it does not classify the file.

READ CODE CAREFULLY

Readable code is guidance, not recovered source.

Available

TALON pseudocode

Use it when: a function is easier to review as pseudocode. Input: disassembly, control flow, imports, and function context. Output: readable-code guidance and confidence notes. Limit: treat it as reconstruction, not exact original source.

Available

Function Intelligence

Use it when: you want a selected-function evidence card. Output: imports, callers, callees, pseudocode, runtime observations when available, and known limits. Limit: advisory function evidence only; GYRE remains verdict authority.

Available

ECHO signatures

Use it when: exact or fuzzy matches may explain a familiar library, pattern, or family clue. Output: signature correlations to review. Limit: a match is a clue, not a magic answer.

WORK FROM EVIDENCE

Group clues before you share them.

Internal tester

STRIKE trace notes

What it is: a runtime-evidence organizer for traces, debugger notes, or observations collected through an approved workflow. Use it when: you already have runtime evidence and want HexHawk to group it into timelines, behaviors, and review notes. Do not use it as: a sandbox, file runner, or automatic malware decision.

Available

NEST evidence groups

What it is: evidence orchestration around GYRE-linked results. Use it when: signals conflict, confidence is low, or you need a clearer handoff package. Output: grouped facts, convergence notes, and source-labelled evidence. Limit: NEST does not become the verdict authority.

Available

CREST reports

Use it when: another person needs to review or repeat your work. Input: file facts, GYRE verdict, NEST evidence, notes, and labelled helper outputs. Output: a report with uncertainty and limits. Limit: a report is a handoff, not independent proof.

WHO DECIDES?

Engine names, translated.

Authority

GYRE decides classification

GYRE is the final classification and base-confidence source. Use it as the starting answer, then inspect the signals and contradictions behind it. Other modules can add evidence or context but must not silently replace GYRE.

Helper

AETHERFRAME improves drafts

AETHERFRAME can suggest clearer wording, safer next steps, or bounded confidence metadata when policy allows. Use it to improve a draft or report. Do not treat it as proof or as a replacement for GYRE/NEST evidence.

Assistant

NEXUS helps the analyst

NEXUS is the assistant layer for plain-English help, workflow suggestions, and reviewed notes. Use it to speed navigation and writing. Do not let it create unreviewed evidence or final security truth.
