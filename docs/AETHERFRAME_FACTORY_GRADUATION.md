# AetherFrame Factory Graduation

Status: migration criteria defined
Date: 2026-07-01

## Purpose

The AetherFrame/Hermes Factory was temporary scaffolding: a disciplined way for Hermes to run bounded improvement cycles against HexHawk while AetherFrame doctrine, lessons, gates, and reporting matured.

Graduation means the Factory has done its job. The durable behavior is absorbed into AetherFrame's normal operating model, and Hermes can work through AetherFrame Advancement Cycles without needing a separate Factory identity.

Graduation does **not** mean uncontrolled automation. AetherFrame remains advisory advancement/refinement/orchestration only. It does not deploy, delete, sign, publish, charge money, or claim truth without explicit approval and evidence.

## Fixed authority boundaries

- GYRE = sole verdict/classification authority.
- NEST = evidence orchestration and convergence.
- TALON = advisory reconstruction only.
- STRIKE = runtime/debugger evidence only.
- Function Intelligence = advisory evidence notebook only.
- AETHERFRAME = advancement/refinement/orchestration only.
- Hermes/AI/NEXUS = assistant/proposal/workflow helper only.

AetherFrame must never become malware verdict authority.

## AetherFrame maturity criteria

- [x] AetherFrame has a clear definition beyond branding: a reusable bounded advancement system.
- [x] AetherFrame can run bounded advancement cycles: intake, boundary check, evidence, research, narrow change, validation, report, lesson, stop.
- [x] AetherFrame can preserve protected fields and authority boundaries in docs and gates.
- [x] AetherFrame can produce evidence reports through the current compatibility reporter.
- [x] AetherFrame can record lessons through an AetherFrame-native lessons ledger.
- [x] AetherFrame can recommend next slices without taking unsafe action.
- [x] AetherFrame can be reused beyond HexHawk in principle through the project template.
- [x] AetherFrame now has a project-neutral runbook/template.
- [x] AetherFrame does not claim autonomous truth, verdict, release, deletion, or deployment authority.
- [~] Reporter tooling still uses the legacy factory filename and should evolve gradually.

## Factory redundancy criteria

- [x] Factory lessons are migrated into AetherFrame-native lessons.
- [x] Factory runbook has an AetherFrame-native replacement.
- [x] New improvement cycles can now be described as AetherFrame Advancement Cycles.
- [~] Factory reporter output is marked as legacy/compatibility but not yet renamed.
- [~] Remaining Factory files are mostly historical continuity, but should remain until the user approves archival/removal.

## Graduation decision states

1. **Not ready** — core docs/gates/lessons do not exist or CI/release posture is unsafe.
2. **Partially ready** — useful behavior exists but Factory still carries core process responsibilities.
3. **Ready to begin migration** — AetherFrame has native docs/templates/lessons/runbooks, but compatibility files remain.
4. **Ready to archive Factory** — AetherFrame reporter/checklists have replaced Factory wording and evidence has been preserved.
5. **Ready to remove Factory scaffolding** — only after explicit user approval and after archival is proven safe.

## Current decision

**Ready to begin migration.**

The Factory should not be destroyed today. AetherFrame is now useful enough for Hermes to prefer AetherFrame Advancement Cycle wording, but old Factory files and the reporter remain historical/compatibility scaffolding until a later approved archive/removal pass.
